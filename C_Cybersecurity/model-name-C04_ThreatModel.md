## 2. 시스템 설명 및 분석 범위

### 2.1 model-name 시스템 개요

model-name Console SW는 디지털 X-ray 영상 검출기(DR Detector)로부터 획득한 의료 영상(DICOM)을 표시·조작하고 PACS(Picture Archiving and Communication System) 및 HIS(Hospital Information System)로 전송하는 **독립형 의료기기 소프트웨어(SaMD)**이다.

**운영 환경**:
- 설치 위치: 병원 방사선실 내 임상용 워크스테이션 (Windows 기반)
- 네트워크: 병원 내부 LAN (인터넷 직접 연결 없음 — 단, 병원 인프라에 따라 상이)
- 사용자: 방사선사(Technician), 방사선과 의사(Radiologist), 시스템 관리자(Admin)

### 2.2 분석 범위

| 범위 구분 | 포함/제외 | 설명 |
|---|---|---|
| model-name Console SW 애플리케이션 | ✅ 포함 | 본 위협 모델의 주 분석 대상 |
| model-name와 DR Detector 간 통신 | ✅ 포함 | DICOM 영상 획득 인터페이스 |
| model-name와 PACS 간 통신 | ✅ 포함 | DICOM Send/Query/Retrieve |
| model-name와 HIS/RIS 간 통신 | ✅ 포함 | HL7 또는 DICOM Worklist |
| 설치 환경 OS (Windows) | ☐ 부분 포함 | OS 취약점은 병원 IT 책임. model-name 설정 영향 범위만 포함 |
| DR Detector 하드웨어 내부 | ❌ 제외 | 별도 기기 — model-name 범위 외 |
| 병원 네트워크 인프라 (스위치, 방화벽) | ❌ 제외 | 병원 IT 운영 책임 |

## 비고: 이 문서가 필요한 이유

### 이 문서가 없으면?

위협 모델은 모든 사이버보안 산출물의 출발점이다. C05(보안 위험 평가), C06(침투 테스트 범위), C07(모니터링 우선순위) 전체가 위협 모델에서 도출된다. 위협 모델 없이 나머지 사이버보안 문서를 작성하면 "무엇을 기반으로 이 위험을 평가했는가?", "무엇을 테스트했는가?"라는 FDA 심사관의 질문에 답할 수 없다. FDA는 위협 모델 없이 제출된 510(k)에 대해 Deficiency Letter를 발송하며, 이는 사이버보안 전체 패키지를 재제출해야 하는 상황으로 이어진다.

- **FDA**: 2026 Guidance §III-B에서 "위협 모델이 사이버보안 설계 결정의 기반이 되어야 한다"고 명시. 위협 모델 없으면 C05~C07 전체가 근거 없는 문서가 되며, 패키지 전체에 대한 deficiency가 발생한다.
- **MFDS**: 현재 명시적으로 요구하지 않으나, FDA 대응 과정에서 작성된 위협 모델은 MFDS 사이버보안 안전성 설명 자료로 활용할 수 있다.
- **EU MDR**: Annex I §17.2 및 MDCG 2019-16은 위협 분석(Threat Analysis)을 사이버보안 설계 프로세스의 필수 단계로 규정한다. Notified Body는 위협 모델을 사이버보안 설계 합리성 평가의 첫 번째 검토 문서로 사용한다.

### 시장별 요구 수준

| 시장 | 요구 수준 | 설명 |
|------|----------|------|
| FDA | 필수 | 2026 Guidance §III-B 직접 요건. 없으면 C05~C07 전체 근거 없어짐. Deficiency Letter 확실 |
| MFDS | 권장 | 공식 의무 없음. FDA 대응 과정에서 작성 후 사이버보안 안전성 설명 자료로 활용 가능 |
| EU MDR | 필수 | MDCG 2019-16 + Annex I §17.2 필수 단계. Notified Body 첫 번째 검토 문서 |

### 최소 필수 vs 리스크 최소화

- **최소 필수로 작성할 경우**: DFD(데이터 흐름도)와 신뢰 경계선(Trust Boundary), STRIDE 방법론을 사용한 위협 식별(스푸핑, 변조, 부인, 정보 유출, 서비스 거부, 권한 상승), 각 위협에 대한 초기 위험 평가(발생 가능성 × 영향도)를 포함한다. SAD(A03)의 아키텍처 다이어그램에 신뢰 경계선을 추가하면 DFD 작성 시간을 크게 줄일 수 있다.
- **리스크 최소화로 확장할 경우**: MITRE ATT&CK for ICS/Medical Devices 프레임워크를 적용하면 산업 특화 공격 패턴을 체계적으로 커버할 수 있다. PASTA 방법론을 추가로 적용하면 비즈니스 위험 관점의 위협까지 포함하여 EU MDR의 요건을 더 완전하게 충족할 수 있다. 보안 전문 기관에 C04와 C06을 동시에 의뢰하면 위협 모델에서 도출된 시나리오가 침투 테스트의 테스트 케이스로 직접 연결되어 비용 효율이 높아진다.

### 외주 개발 시 유의사항

위협 모델은 "공격자 관점에서 시스템을 바라보는" 전문 역량이 필요하다. 내부에 사이버보안 전문가가 없다면 외부 전문기관에 위탁하는 것이 가장 현실적인 방법이다. SOW에 다음을 명시할 것: ① 외주 개발사는 DFD 작성을 위한 시스템 아키텍처 정보(SAD A03 기반)를 사이버보안 전문기관에 제공하는 지원 의무, ② 위협 모델 작성은 사이버보안 전문기관 또는 company-name 내부 전담자가 수행(개발 외주사가 아님), ③ 위협 모델 완성 후 외주 개발사는 식별된 위협에 대응하는 보안 통제(C03)를 구현하는 의무. 위협 모델은 개발 착수 초기(아키텍처 설계 완료 후 즉시)에 작성해야 하며, 개발 완료 후 작성하면 이미 완성된 코드에서 발견된 위협을 수정하는 비용이 배로 증가한다.
