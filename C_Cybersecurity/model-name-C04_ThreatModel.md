## 2. 시스템 설명 및 분석 범위

### 2.1 model-name 시스템 개요

model-name Console SW는 디지털 X-ray 영상 검출기(DR Detector)로부터 획득한 의료 영상(DICOM)을 표시·조작하고 PACS(Picture Archiving and Communication System) 및 HIS(Hospital Information System)로 전송하는 **독립형 의료기기 소프트웨어(SaMD)**이다.

**운영 환경**:
- 설치 위치: 병원 방사선실 내 임상용 워크스테이션 (Windows 기반)
- 네트워크: 병원 내부 LAN (인터넷 직접 연결 없음 — 단, 병원 인프라에 따라 상이)
- 사용자: 방사선사(Technician), 방사선과 의사(Radiologist), 시스템 관리자(Admin)

### 2.2 분석 범위

| 범위 구분 | 포함/제외 | 설명 |
|---|---|---|
| model-name Console SW 애플리케이션 | ✅ 포함 | 본 위협 모델의 주 분석 대상 |
| model-name와 DR Detector 간 통신 | ✅ 포함 | DICOM 영상 획득 인터페이스 |
| model-name와 PACS 간 통신 | ✅ 포함 | DICOM Send/Query/Retrieve |
| model-name와 HIS/RIS 간 통신 | ✅ 포함 | HL7 또는 DICOM Worklist |
| 설치 환경 OS (Windows) | ☐ 부분 포함 | OS 취약점은 병원 IT 책임. model-name 설정 영향 범위만 포함 |
| DR Detector 하드웨어 내부 | ❌ 제외 | 별도 기기 — model-name 범위 외 |
| 병원 네트워크 인프라 (스위치, 방화벽) | ❌ 제외 | 병원 IT 운영 책임 |

## 비고: 왜 이 문서가 필요한가 (개발팀·경영진을 위한 설명)

### ✅ 최소 필수 선택 근거

**FDA 직접 요구**  
FDA Cybersecurity Guidance Feb 2026 §III-B는 "위협 모델이 사이버보안 설계 결정의 기반이 되어야 한다"고 명시한다. 위협 모델 없이 510(k) 제출 시 FDA는 "어떤 위협을 고려했는가"에 대한 해명을 요구하는 Deficiency Letter를 발송한다. 사실상 제출 불가 상태로 이어진다.

**EU MDR 직접 요구**  
EU MDR Annex I §17.2 및 MDCG 2019-16은 위협 분석(Threat Analysis)을 사이버보안 설계 프로세스의 필수 단계로 규정한다.

**model-name-C05 (보안 위험 평가)의 입력 문서**  
위협 모델 없이는 보안 위험 평가를 작성할 수 없다. 위협 모델이 먼저 완성되어야 이후 모든 보안 산출물(C05, C06 범위 정의, C07 모니터링 우선순위)이 작성 가능하다.

### 개발팀 실무 가이드

**외부 전문기관 활용 강력 권장**:  
위협 모델은 "공격자 관점에서 시스템을 보는" 능력이 필요하다. 내부에 사이버보안 전문가가 없다면 외부 전문기관에 위협 모델 및 침투 테스트(C06)를 동시에 의뢰하는 것이 비용 효율적이다.

**DFD가 핵심**:  
위협 모델의 품질은 DFD(데이터 흐름도)의 정확성에 달려 있다. SAD(model-name-A03)의 아키텍처를 그대로 가져와 신뢰 경계선을 추가하면 된다. DFD가 정확할수록 STRIDE 분석의 빠짐이 줄어든다.

**STRIDE vs 다른 방법론**:  
STRIDE는 가장 보편적이고 FDA에서 수용되는 방법론이다. model-name와 같은 소규모 SaMD에는 STRIDE가 적합하다. 더 복잡한 시스템이나 EU 요건 강화 시 PASTA 또는 MITRE ATT&CK for Medical Devices 활용을 검토할 수 있다.
