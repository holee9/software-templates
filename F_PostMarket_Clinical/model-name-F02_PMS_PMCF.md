### 1.2 PMS 목표

EU MDR Article 83에 따라 다음 목표를 달성하기 위한 PMS 시스템을 수립한다:

1. model-name의 안전성, 성능 및 허용 가능한 품질 수준을 지속적으로 확인
2. 잔여 위험 수용 가능성 지속 검토 (model-name-D01 연계)
3. CER(model-name-F01)을 갱신하기 위한 임상 데이터 지속 수집
4. 이상사례 및 FSCA(Field Safety Corrective Action)를 조기에 식별하고 적절히 대응
5. 사이버보안 취약점 발생 시 신속한 패치 및 사용자 통지

### 1.3 PMS 데이터 수집 방법

### 1.4 이상사례 보고 기준 및 절차

#### 보고 기한 (EU MDR Article 87)

| 이상사례 유형 | 보고 기한 | 보고 대상 |
|---|---|---|
| 심각한 위협 (공중 보건 위협) | 2일 이내 | 각 EU 회원국 관할 기관 |
| 사망 또는 예상치 못한 심각한 위해 | 10일 이내 | 각 EU 회원국 관할 기관 |
| 심각한 공중 위해 위험 외 이상사례 | 15일 이내 | 각 EU 회원국 관할 기관 |
| 추세 보고 (Trend Reporting) | 30일 이내 | 각 EU 회원국 관할 기관 |
| 필드 안전 교정 조치 (FSCA) 전 고지 | 사전 통지 (가능한 경우) | EUDAMED + 각국 기관 |

#### 이상사례 처리 절차

```
접수 → 초기 평가 (24시간) → 심각도 분류 → 
규제 기관 보고 여부 결정 → 보고 (기한 내) → 
조사 및 CAPA → 위험 관리 파일 업데이트 → 
CER 갱신 여부 검토
```

### 1.5 FSCA (Field Safety Corrective Action) 및 사이버보안 패치

#### 패치 분류 기준 (FDA vs EU MDR 비교)

| 패치 분류 | CVSS 기준 | FDA §524B 대응 | EU MDR FSCA 해당 여부 | SLA |
|---|---|---|---|---|
| **Critical** | ≥ 9.0 | 즉시 보고 필요 (§524B) | ⚠️ FSCA 해당 가능 (임상 위험 평가 필요) | 배포 14일, 적용 즉시 권고 |
| **High** | 7.0~8.9 | 30일 내 조치 계획 | 임상 영향 평가 후 FSCA 여부 판단 | 배포 30일, 적용 14일 |
| **Medium** | 4.0~6.9 | 정기 업데이트 포함 | 해당 없음 (일반적) | 배포 90일, 정기 업데이트 |
| **Low** | < 4.0 | 다음 정기 릴리즈 | 해당 없음 | 다음 정기 릴리즈 |

#### FSCA가 필요한 사이버보안 사고 예시

| 사고 유형 | FSCA 해당 여부 | 조치 방향 |
|---|---|---|
| 영상 변조를 유발할 수 있는 Critical 취약점 발견 | ✅ FSCA 필요 | 즉시 패치 배포 + 사용자 통지 + 규제 기관 보고 |
| 인증 우회 취약점 (CVSS ≥ 7.0) | ⚠️ 임상 위험 평가 후 판단 | 위험 평가 → FSCA 여부 결정 → 패치 배포 |
| 알려진 SOUP 취약점 (Non-exploitable) | ❌ FSCA 불필요 | VEX 업데이트, 다음 정기 패치 포함 |

## 4. 리콜 절차 (Recall Procedure)

*EU MDR Article 83(3)(e) 준수*

### 4.1 리콜 분류 기준

| 분류 | 정의 | 예시 |
|---|---|---|
| Class I Recall | 중대한 건강 위해 또는 사망 가능성 | 영상 변조를 유발하는 사이버보안 공격 취약점 |
| Class II Recall | 일시적 또는 가역적 건강 문제 가능성 | 영상 표시 오류를 유발하는 SW 버그 |
| Class III Recall | 건강 위해 가능성 없음, 법규 위반 | UDI 표시 오류, 라벨링 오류 |
| Market Withdrawal | 제조 결함 없음, 유통 문제 | 라이선스 오류, 비의도적 배포 |

### 4.2 리콜 절차 흐름

```
문제 식별 → 초기 위험 평가 (24시간) → 리콜 필요성 판단 →
경영진 승인 → 규제 기관 사전 통지 (EUDAMED + 각국 기관) →
고객/사용자 통지 → 제품 수거/수정 배포 → 효과 검증 →
위험 관리 파일 업데이트 → 최종 리콜 보고 →
재발 방지 CAPA
```

### 4.3 SW 특화 리콜 조치 (Forced Update / Remote Patch)

| 조치 유형 | 내용 | 해당 상황 |
|---|---|---|
| 긴급 강제 업데이트 | 다음 기동 시 자동 업데이트 강제 (선택 불가) | Class I 리콜 수준 Critical 취약점 |
| 권고 업데이트 | 업데이트 알림 + 강력 권고 | Class II 수준 |
| 원격 기능 비활성화 | 취약한 기능을 원격으로 비활성화 (해당 기능 구현 시) | 긴급 상황 |

## 비고: 리스크 최소화 선택 근거 (개발팀 안내)

**왜 "리스크 최소화"로 분류했는가? (EU MDR에서 사실상 필수인 이유)**

PMS/PMCF 문서는 FDA와 MFDS에서 요구 수준이 낮기 때문에 "리스크 최소화"로 분류되었다. 그러나 EU MDR CE 마킹을 추진하는 Plan B 전략에서는 절대적으로 필수이다. EU MDR은 시판 후 감시를 허가 유지의 핵심 조건으로 규정한다.

**EU MDR 시판 후 의무의 엄격성**:
- PSUR 미제출 시 Notified Body가 CE 마킹 정지(Suspension) 조치 가능
- PMCF 보고서 미갱신 시 중대 결함(Major Non-conformity) 지적
- 이상사례 보고 기한 미준수 시 관할 규제 기관의 행정 처분 가능

**한국 기업이 특히 주의해야 할 사항**: 한국 기업이 EU에 진출할 때 허가는 받아도 시판 후 의무(PMS, PMCF, PSUR)를 지속적으로 이행하지 않아 CE 마킹을 정지당하는 사례가 많다. 내부 RA/QA 인력이 EU MDR 시판 후 의무를 숙지하고, 달력에 PSUR 제출 일정을 미리 등록해야 한다.

**PMCF vs PMS 차이 (개발팀 안내)**:
- **PMS**: 모든 의료기기에 적용되는 넓은 개념의 시판 후 감시. 불만, 이상사례, 문헌 모니터링 등 포함
- **PMCF**: EU MDR 특화. 임상 데이터를 지속적으로 수집하여 CER을 갱신하는 임상 특화 활동. Class IIa 이상 의무

**소규모 기업 실용 팁**: 별도 임상 시험 없이 실제 고객 병원의 익명화된 사용 데이터를 수집하는 Registry-based PMCF가 비용 효율적이다. 이를 위해 고객사와 PMCF 데이터 활용 동의 계약을 체결하는 것이 핵심이다.

**사이버보안과 FSCA 연계 주의**: Critical 취약점 발생 시 EU MDR FSCA 대상이 될 수 있다. 취약점 패치를 "단순 SW 업데이트"로 처리하려는 경향이 있으나, 임상 안전성에 영향을 미치는 취약점은 반드시 위험 평가 후 FSCA 여부를 판단해야 한다.
