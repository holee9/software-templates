## 1. 목적

본 문서는 model-name Console SW에 포함된 제3자 라이브러리, 오픈소스 컴포넌트, 상용 SW(COTS)를 목록화하고 각 SOUP의 위험을 평가한다.

라이선스 컴플라이언스, 알려진 취약점 관리, 버전 고정 정책의 근거 문서로 활용된다.

### 4.6 SOUP 업데이트 관리 정책

1. **정기 검토**: 분기 1회 각 SOUP의 신규 CVE 및 버전 업데이트 확인
2. **긴급 패치**: Critical/High CVE 발생 시 30일 이내 패치 적용 또는 완화 조치
3. **버전 변경 절차**:
   - 변경 요청(CR) 생성 (model-name-A05 변경 통제 프로세스)
   - SOUP 위험 재평가
   - 회귀 시험 수행
   - 형상관리 시스템 업데이트
   - 본 문서 (model-name-A04) 개정
4. **허가 영향 평가**: SOUP 주요 버전 변경 시 FDA/MFDS 규제 변경 보고 여부 RA 판단

## 5. 비고: 최소 필수 vs 리스크 최소화 선택 가이드

**SOUP 목록 vs SBOM의 차이**:  
SOUP 목록(본 문서)은 IEC 62304 §8.1 규제 요건으로, 각 컴포넌트의 **위험 평가 및 검증** 기록에 초점을 맞춘다. SBOM(model-name-C01)은 FDA §524B 사이버보안 요건으로, 모든 컴포넌트의 **기계 판독 가능한 목록**에 초점을 맞춘다. 두 문서는 동일한 데이터를 기반으로 하지만 목적이 다르므로 별도 유지하며, SOUP 목록에서 SBOM을 파생 생성하는 것이 효율적이다.

**없으면 어떤 리스크?**  
- **FDA**: SOUP 미관리 → SW 안전성 증명 불가. eSTAR 심사에서 필수 확인 항목  
- **MFDS**: 제3자 라이브러리 목록 및 검증 근거 제출 요구  
- **EU MDR**: SOUP에 대한 위험 평가 미실시 시 Annex I §17.4 부적합  

**업데이트 의무**: SW 버전이 변경될 때마다(SOUP 버전 포함) 본 문서를 반드시 업데이트해야 한다. 형상관리 시스템(model-name-A05)과 연동하여 자동 알림 설정 권장.
