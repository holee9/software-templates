## 1. 목적

본 문서는 model-name Console SW의 허가 제출 시점 또는 각 릴리즈 시점에서 알려져 있지만 해결되지 않은 SW 결함(버그, 이상 동작)을 목록화한다.  
각 결함의 임상적·안전적 영향과 허용 근거를 명시하여, 규제 기관이 잔여 위험을 평가할 수 있도록 투명하게 공개한다.  
알려진 결함이 없는 경우에도 "알려진 결함 없음"을 명시적으로 기재하여 제출한다.

## 4. 본문

### 4.1 결함 분류 기준

#### 4.1.1 심각도 분류

| 심각도 | 정의 | 예시 |
|--------|------|------|
| Critical | 환자 안전에 직접 위해를 가할 수 있는 결함 | 영상 환자 ID 오표시, 측정값 오류 |
| Major | 핵심 기능 사용 불가 또는 심각한 성능 저하 | PACS 전송 실패, 영상 표시 불가 |
| Moderate | 기능 일부 제한, 대안(Workaround) 존재 | 특정 조건에서 UI 오류, 성능 저하 |
| Minor | 기능에 영향 없는 외관상 결함, 불편 | 오타, UI 정렬 문제, 경미한 성능 |

#### 4.1.2 허용 기준

결함이 아래 조건을 **모두** 충족하는 경우에만 허용(미해결 릴리즈)이 가능하다:

1. 환자 안전(Safety)에 직접 또는 간접 위해가 없다
2. 진단 정확도에 영향을 미치지 않는다
3. 위험-편익 분석에서 잔여 위험이 허용 기준 이내이다 (model-name-D01 참조)
4. QA 및 RA 책임자가 서면으로 허용을 승인한다

> **Critical 결함**: Critical 심각도 결함은 어떠한 경우에도 허용할 수 없으며, 해결 전까지 릴리즈가 금지된다.

## 5. 비고: 최소 필수 vs 리스크 최소화 선택 가이드

**왜 최소 필수인가?**  
FDA는 "Revision History and Anomaly List"를 SW Documentation의 필수 구성 요소로 요구한다. 알려진 결함을 숨기거나 문서화하지 않으면 허가 심사 과정에서 더 큰 문제가 된다. MFDS와 EU MDR도 동등하게 잔여 결함 공개를 요구한다.

**없으면 어떤 리스크?**  
- **FDA**: eSTAR Section 12 미충족 → AI (Additional Information) 요청 확실  
- **MFDS**: 소프트웨어 이력 관리 미흡 → 허가 반려 가능성  
- **EU MDR**: Annex I §17.2 "SW의 알려진 한계" 미공개 → 부적합 판정  

**"결함 없음"도 제출 필수**:  
알려진 결함이 전혀 없더라도 본 문서를 "현재 알려진 결함 없음"으로 작성하여 제출해야 한다. 빈 목록은 오히려 심사관에게 신뢰를 준다. 빈 칸으로 두거나 문서를 생략하면 미검토로 간주된다.

**FDA 실무 팁**:  
경미한 결함(Minor)이 수십 개 있어도 임상 영향이 없음을 명확히 기술하면 허가에 문제없다. 모든 결함에 대해 "안전 영향 평가"와 "완화 조치"를 기재하면 심사관의 신뢰를 높인다. **숨기는 것이 더 위험하다.**
