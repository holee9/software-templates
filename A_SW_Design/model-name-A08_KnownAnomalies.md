## 1. 목적

본 문서는 model-name Console SW의 허가 제출 시점 또는 각 릴리즈 시점에서 알려져 있지만 해결되지 않은 SW 결함(버그, 이상 동작)을 목록화한다.  
각 결함의 임상적·안전적 영향과 허용 근거를 명시하여, 규제 기관이 잔여 위험을 평가할 수 있도록 투명하게 공개한다.  
알려진 결함이 없는 경우에도 "알려진 결함 없음"을 명시적으로 기재하여 제출한다.

## 4. 본문

### 4.1 결함 분류 기준

#### 4.1.1 심각도 분류

| 심각도 | 정의 | 예시 |
|--------|------|------|
| Critical | 환자 안전에 직접 위해를 가할 수 있는 결함 | 영상 환자 ID 오표시, 측정값 오류 |
| Major | 핵심 기능 사용 불가 또는 심각한 성능 저하 | PACS 전송 실패, 영상 표시 불가 |
| Moderate | 기능 일부 제한, 대안(Workaround) 존재 | 특정 조건에서 UI 오류, 성능 저하 |
| Minor | 기능에 영향 없는 외관상 결함, 불편 | 오타, UI 정렬 문제, 경미한 성능 |

#### 4.1.2 허용 기준

결함이 아래 조건을 **모두** 충족하는 경우에만 허용(미해결 릴리즈)이 가능하다:

1. 환자 안전(Safety)에 직접 또는 간접 위해가 없다
2. 진단 정확도에 영향을 미치지 않는다
3. 위험-편익 분석에서 잔여 위험이 허용 기준 이내이다 (model-name-D01 참조)
4. QA 및 RA 책임자가 서면으로 허용을 승인한다

> **Critical 결함**: Critical 심각도 결함은 어떠한 경우에도 허용할 수 없으며, 해결 전까지 릴리즈가 금지된다.

## 비고: 이 문서가 필요한 이유

### 이 문서가 없으면?

FDA는 "알려진 결함이 없다"는 주장을 믿지 않는다. 알려진 결함 목록이 없으면 "검토하지 않은 것"으로 간주되며, 오히려 숨기고 있다는 의심을 받는다. FDA 심사관이 eSTAR를 검토할 때 Known Anomalies 목록이 없으면 즉시 AI(Additional Information) 요청을 보낸다. 반대로, 경미한 결함이 수십 개 있어도 각각에 대해 안전 영향 평가와 완화 조치를 기술하면 허가에 문제가 없다. 투명성이 신뢰를 만든다.

- **FDA**: "Revision History and Anomaly List"는 IEC 62304 §9.8과 FDA SW Documentation Guidance의 필수 구성 요소다. 없으면 eSTAR Section 12 미충족으로 AI 요청이 확실히 발송된다. 결함 목록 없이 "결함 없음"을 주장하면 FDA 심사관의 신뢰를 잃는다. 결함이 없더라도 "현재 알려진 결함 없음 (No Known Anomalies)"을 명시적으로 기재해야 한다.
- **MFDS**: 소프트웨어 이력 관리 요건 하에 잔여 결함 공개를 요구한다. 결함 목록이 없으면 "소프트웨어 검증이 충분히 이루어지지 않았다"는 판단의 근거가 된다.
- **EU MDR**: Annex I §17.2는 "소프트웨어의 알려진 한계"를 공개하도록 요구한다. 결함 목록 미공개는 IFU(사용 설명서)의 금기 사항·경고 기재 의무와도 연결되어 부적합 판정을 초래한다.

### 시장별 요구 수준

| 시장 | 요구 수준 | 설명 |
|------|----------|------|
| FDA | 필수 | eSTAR Section 12 필수 항목. 없으면 AI 요청 확실. "결함 없음"도 명시적 기재 필요 |
| MFDS | 필수 | SW 이력 관리 요건. 미공개 시 검증 불충분 판단 근거가 됨 |
| EU MDR | 필수 | Annex I §17.2 직접 요건. IFU 기재 의무와 연계. 미공개 시 부적합 판정 |

### 최소 필수 vs 리스크 최소화

- **최소 필수로 작성할 경우**: 결함 ID, 결함 요약, 재현 방법, 심각도(Critical/Major/Minor), 환자 안전 영향 평가, 완화 조치(workaround 또는 사용 제한 사항), 해결 버전(있는 경우)을 포함한다. 결함이 없는 경우에도 "현재 알려진 결함 없음"을 명시적으로 기재하고 QA 책임자가 서명해야 한다.
- **리스크 최소화로 확장할 경우**: 각 결함의 환자 안전 영향을 ISO 14971 위험 관리 파일(D01)과 연동하면 "이 결함이 허용 가능한 위험 수준에 있다"는 근거가 명확해진다. 결함 추적 시스템(Jira, GitHub Issues 등)의 버그 ID를 본 문서와 연동하면 FDA 감사 시 "이 결함이 실제로 인지되고 추적되었음"을 즉시 증명할 수 있다. IFU(사용 설명서)의 경고/주의 문구와 결함 완화 조치를 연계하면 EU MDR Annex I 요건을 효율적으로 충족할 수 있다.

### 외주 개발 시 유의사항

외주 개발 종료 시 외주사로부터 반드시 "알려진 결함 목록"을 납품 산출물에 포함하도록 SOW에 명시해야 한다. 실무적으로 외주사는 결함을 숨기거나 축소 기재하는 경향이 있으므로, 결함 추적 시스템(Jira 등)에 대한 company-name의 접근 권한을 계약 기간 내 확보해야 한다. SOW에 다음을 명시할 것: ① 납품 시 Known Anomalies 목록 제출 의무(결함이 없어도 "결함 없음" 확인서 포함), ② company-name의 QA 담당자가 결함 추적 시스템을 직접 열람할 수 있는 접근 권한 부여, ③ 인수 시험(Acceptance Test) 중 발견된 결함의 처리 계획 제출 의무.
