### 4.3 버전 관리 정책 (VCS Policy)

#### 4.3.1 버전 번호 체계

```
MAJOR.MINOR.PATCH
│      │     └── 버그 수정 (하위 호환)
│      └──────── 기능 추가 (하위 호환)
└─────────────── 비호환 변경 / 주요 기능 개편
```

예시: `1.2.3` = 주 버전 1, 부 버전 2, 패치 버전 3

#### 4.3.2 브랜칭 전략 (Git Flow 기반)

| 브랜치 | 목적 | 보호 설정 | 병합 대상 |
|--------|------|----------|----------|
| `main` | 릴리즈 완료 코드 (허가 버전) | 직접 Push 금지, PR + 2인 승인 | `release/*` 에서 병합 |
| `develop` | 개발 통합 브랜치 | PR + 1인 승인 | `feature/*` 에서 병합 |
| `feature/[기능명]` | 신규 기능 개발 | 없음 | `develop` 으로 병합 |
| `hotfix/[이슈ID]` | 긴급 버그 수정 | 없음 | `main` 및 `develop` 으로 병합 |
| `release/[버전]` | 릴리즈 준비 (최종 버그 수정) | PR + 2인 승인 | `main` 및 `develop` 으로 병합 |

#### 4.3.3 커밋 메시지 규칙

```
[유형]: [요약] (#이슈번호)

유형: feat | fix | docs | test | refactor | chore | security
예: feat: DICOM C-STORE TLS 1.3 지원 추가 (#123)
    fix: 영상 표시 타임아웃 오류 수정 (#456)
```

#### 4.3.4 릴리즈 태그 정책

- 형식: `v[MAJOR.MINOR.PATCH]` (예: `v1.0.0`)
- 태그 생성 시점: `release/*` 브랜치가 `main`에 병합된 후
- 태그 서명: GPG 서명 적용 (권장) 또는 SHA-256 기록
- 태그 삭제: 금지 (과거 릴리즈 버전 접근 가능성 보장)

### 4.4 빌드 프로세스

#### 4.4.1 CI/CD 파이프라인 개요

```
[코드 Push / PR 생성]
    → [정적 분석] (SonarQube 등)
    → [단위 테스트 실행]
    → [빌드 (MSBuild/CMake)]
    → [통합 테스트 (선택)]
    → [빌드 산출물 서명]
    → [아티팩트 저장소 업로드]
```

#### 4.4.2 릴리즈 빌드 절차

1. `release/[버전]` 브랜치 생성
2. 버전 번호 업데이트 (소스코드 내 버전 상수, 설치 패키지 버전)
3. 최종 버그 수정 및 코드 리뷰
4. 릴리즈 빌드 실행 (재현 가능한 빌드 환경 — model-name-A07 참조)
5. 빌드 산출물 SHA-256 해시 생성 및 기록
6. 코드 서명 적용 (MSI 설치 패키지)
7. QA 시험 통과 확인
8. `main` 병합 및 태그 생성
9. 릴리즈 기록(model-name-A06) 생성

#### 4.4.3 빌드 재현성 보장

- 빌드 환경은 Docker 이미지 또는 VM 스냅샷으로 고정 (상세: model-name-A07)
- 모든 의존성(SOUP)은 아티팩트 저장소에서 버전 고정 참조
- 빌드 환경 변경 시 새 이미지/스냅샷 생성 및 이전 이미지 보존

### 4.5 변경 통제 프로세스 (Change Control)

#### 4.5.1 변경 요청 (Change Request, CR) 절차

```
1. CR 생성
   - 이슈 추적 시스템에 CR 티켓 생성
   - CR 유형: Bug Fix / Feature / Security Patch / Documentation

2. 영향도 평가 (IEC 62304 §6.2.5 준수)
   - 기능 영향: 어느 컴포넌트에 영향을 미치는가?
   - 안전 영향: 환자 안전 관련 기능에 영향이 있는가?
   - 규제 영향: FDA/MFDS/EU MDR 보고 여부 판단
   - SOUP 영향: SOUP 버전 변경 필요 여부

3. 승인
   - Minor: 개발 리더 승인
   - Major: PM + QA 책임자 승인
   - Safety-related: PM + QA + RA 승인

4. 구현 및 시험
   - 구현 (feature/* 브랜치)
   - 코드 리뷰 + 회귀 시험

5. 베이스라인 업데이트
   - 관련 문서 개정 (SRS, SAD 등)
   - 형상관리 시스템 업데이트
```

#### 4.5.2 변경 유형별 승인 권한

| 변경 유형 | 정의 | 승인 권한 | 규제 보고 |
|----------|------|----------|----------|
| Minor | 기능 영향 없는 버그 수정, UI 텍스트 | 개발 리더 | 내부 기록만 |
| Major | 새 기능, 성능 변경, SOUP 버전 업 | PM + QA | MFDS 경미한 변경 신고 검토 |
| Critical (Safety) | 안전 관련 기능 변경, 적응증 변경 | PM + QA + RA + 대표이사 | FDA/MFDS 변경 허가 검토 |
| Emergency (Hotfix) | 즉시 패치 필요한 보안/안전 취약점 | QA + RA (사후 PM 보고) | 규제 보고 여부 즉시 판단 |

## 5. 비고: 최소 필수 vs 리스크 최소화 선택 가이드

**왜 최소 필수인가?**  
FDA는 "허가된 SW 버전을 언제든 재현할 수 있음"을 요구한다. 형상 관리 계획 없이는 "어떤 버전이 FDA에 제출된 버전인가"를 증명할 수 없어, 허가 후 감사(GMP Inspection) 시 심각한 부적합 사유가 된다.

**없으면 어떤 리스크?**  
- **FDA**: 허가 후 SW 버전 추적 불가 → 품질 시스템 감사 실패  
- **MFDS**: 변경 허가 신청 시 이전 버전과의 비교 불가  
- **EU MDR**: QMS(ISO 13485) 요건 위반 — 형상관리 없이는 ISO 13485 인증 유지 불가  

**규모에 맞는 형상 관리**:  
스타트업 규모라면 GitHub 저장소 + 릴리즈 태그 + CHANGELOG.md로도 기본 요건 충족 가능하다. 중요한 것은 도구의 복잡성이 아니라 **"언제든 허가 버전을 재현할 수 있는 프로세스"** 가 있음을 문서로 증명하는 것이다. 본 계획서에 실제 사용 중인 도구와 절차를 명시하면 충분하다.
